AI 辅助博客写作工作流

从 Skill 设计 → 流程可视化 → 文档输出 → 配图生成，一套完整流程

AI 使用说明

本文主题：用 AI 辅助写博客的完整 SOP，覆盖从 Skill 分析到配图生成的四个步骤

适合提问的方向：

  • 如何用 AI 帮我写一篇工具/流程类的技术博客？
  • blog-workflow 和 ai-first-blog 是什么关系？
  • 如何把一个 Skill 的设计逻辑可视化成流程图？
  • 如何生成一张适合插入博客的流程图截图？

局限性说明：

  • 需要本地能运行 Playwright 截图
  • 需要启动 HTTP 服务（python -m http.server）
  • 博客配图设计依赖 frontend-design Skill 的生成质量

一、工作流总览

最终产物：一篇结构完整的博客草稿 + 配图，人工润色后即可发布。

二、第一步：读取 Skill 源文件

做什么

用 Skill 工具加载目标 Skill，同时读取项目目录下的相关文件（HTML、JSON、MD 等），快速建立对 Skill 的完整认知。

关键问题清单

阅读 Skill 时，重点回答以下问题，这些将成为博客的骨架：

问题 对应博客章节
这个 Skill 解决了什么问题？ 背景与动机
核心设计思路是什么？ 核心设计
有哪几条主要执行路径？ 流程拆解
关键的技术决策是什么？（为什么这样做） 设计亮点
还有哪些未解决的问题？ 遗留问题 / 展望

命令示例

三、第二步：用前端生成流程图 HTML

做什么

调用 frontend-design Skill，将 Skill 的执行逻辑生成一张可交互的 HTML 流程图，作为博客的核心配图。

设计原则

  • 统一入口：所有分支从同一个”用户触发”节点出发，体现 AI 统一决策的思路
  • 颜色语义化：用颜色区分节点类型（缓存/执行/判断/成功/失败），一眼可读
  • 主次分明：最核心的流程（如缓存机制）作为主列，其他作为次要列
  • 共享终点：所有分支汇聚到同一个”返回结果”节点，强调统一出口

迭代建议

流程图通常需要 2-3 轮迭代：

轮次 问题 调整方向
第 1 稿 分支过多，主次不清 合并次要分支，突出核心路径
第 2 稿 各流程彼此孤立 统一入口和出口，体现 AI 决策层
第 3 稿 细节调整 颜色、字号、间距微调

Prompt 模板

四、第三步：HTML 截图为 PNG

做什么

用 playwright-cli Skill 将 HTML 渲染为 PNG，供博客直接插入。

完整步骤

注意事项

  • **为什么不能直接用 file://**：playwright-cli 默认屏蔽本地文件协议，需起 HTTP 服务中转
  • 视口高度：必须先查询 scrollHeight 再设置视口，否则内容被截断
  • 分辨率：1200px 宽适合大多数博客平台；如需高清可用 --scale=2 生成 2x 图

五、第四步：输出博客参考文档

做什么

基于前几步积累的信息，生成一份结构化的 Markdown 文档，作为写博客的原材料。

推荐文档结构

Prompt 模板

六、产物清单

文件 用途
[skill]-flowchart.html 可交互流程图（源文件，可持续维护）
[skill]-flowchart.png 博客配图（直接插入）
[skill]-design.md 博客参考文档（人工润色后发布）

七、适用场景扩展

这套工作流不只适用于 Skill 设计博客，同样适用于：

  • 系统架构博客：把架构文档 → 架构图 → 博客
  • API 设计博客：把接口文档 → 调用流程图 → 博客
  • 工具使用教程：把操作步骤 → 流程图 → 图文教程
  • 代码库解析：把源码阅读笔记 → 模块关系图 → 技术分析文章

核心思路不变：先理解 → 再可视化 → 再文字化。
