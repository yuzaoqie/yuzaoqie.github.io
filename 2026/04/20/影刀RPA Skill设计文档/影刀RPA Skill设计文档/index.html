影刀RPA Skill 设计文档

版本：v1.0 · 2026-04-20
适用对象：Claude Code Skill 开发者 / AI Agent 设计者

AI 使用说明

本文主题：影刀RPA Skill 的完整设计思路，涵盖缓存机制、三条操作流程、技术权衡

适合提问的方向：

  • 如何设计一个能操作影刀RPA应用的 Skill？
  • 缓存优先的查询机制是怎么工作的？
  • 如何让 AI 自动找到应用的 UUID 而不用用户每次提供？
  • 三条流程（查询/启动/状态）是如何设计的？

局限性说明：

  • 不涉及影刀应用的创建和编辑，只涉及启动和查询
  • 缓存失效策略尚未完善（见遗留问题）
  • 用户 ID 目前需要手动配置，不支持自动检测

一、背景与动机

1.1 影刀RPA 是什么

影刀（ShadowBot）是一款本地部署的 RPA（Robotic Process Automation，机器人流程自动化）软件。用户在影刀中创建自动化脚本应用，每个应用拥有一个唯一的 UUID，通过命令行协议启动：

1.2 为什么需要这个 Skill

在没有 Skill 的情况下，用户每次想让 AI 操作影刀，都需要手动提供：

  • ShadowBot.exe 的完整路径
  • 目标应用的 UUID
  • 操作类型（启动 / 查询 / 检查状态）

这不仅繁琐，也打断了自然语言交互的流畅性。

设计目标：用户只需说”帮我启动 XXX 应用”，AI 自动找到 UUID、构造命令、执行并反馈结果。

2.3 缓存设计的核心权衡

方案 优点 缺点
每次实时扫描 数据最新 慢，I/O 多
纯缓存 快 用户新建应用后不感知
缓存 + 按需扫描（本方案） 快且可靠 缓存可能短暂落后

选择第三种方案：命中缓存直接返回，缓存未命中时扫描并更新。这样用户新建应用后，首次查询触发一次扫描，后续均走缓存。

3.1 流程总览

3.2 流程一：查询应用

触发条件：用户问”有哪些影刀应用”、”帮我找 XXX 应用的 UUID”。

执行步骤：

  1. 读取缓存文档 yingdao-apps.md
  2. 检索目标应用名称
  3. 命中 → 直接返回 UUID、名称、说明
  4. 未命中 → 扫描 AppData/.../apps/ 目录 → 解析 package.json → 更新缓存 → 返回

3.3 流程二：启动应用

触发条件：用户说”启动 XXX”、”运行 XXX 应用”。

执行步骤：

  1. 从缓存获取目标应用 UUID（或用户直接提供）
  2. 构造并执行启动命令：
  3. 成功 → 报告”应用已启动，正在运行中”
  4. 失败 → 进入故障排查：核查 EXE 路径是否正确、UUID 是否有效

设计说明：启动命令使用影刀自定义协议 shadowbot:Run?，无需等待进程退出，属于触发型调用。

3.4 流程三：状态检查

触发条件：用户问”影刀在运行吗”、”当前哪些影刀进程在跑”。

执行步骤：

  1. 执行 tasklist | grep -i shadow
  2. 识别关键进程：
进程名 含义
ShadowBot.Shell.exe 影刀主程序
ShadowBot.Bridge.exe 桥接服务
ShadowBot.ChromeBridge.exe Chrome 浏览器桥接
ShadowBot.UIAutomation.Pr UI 自动化引擎
  1. 存在 → “影刀正在运行”
  2. 不存在 → “影刀未启动，建议手动打开主程序”

五、设计亮点总结

5.1 对用户透明的两层存储

用户不需要知道缓存的存在——AI 在背后自动维护缓存与文件系统的一致性，对用户呈现的始终是”我帮你找到了”。

5.2 自然语言 → 精确命令

用户说”启动报表应用”，Skill 完成的事情是：

整个过程对用户完全透明。

5.3 容错与可扩展性

  • 缓存未命中时自动扫描，新应用无需手动注册
  • 故障排查有明确路径（路径错误 / UUID 错误）
  • 支持多用户 ID，切换账户时修改扫描目录即可

六、遗留问题 / 可改进方向

  1. 缓存失效策略：当前缓存永不主动失效，若用户删除了某个应用，缓存中仍有记录。可考虑增加”同步缓存”指令或按时间戳定期全量刷新。
  2. 多用户 ID 自动检测：当前用户 ID 硬编码，可改为自动枚举 users/ 目录下所有子文件夹。
  3. 启动结果验证：命令发出后可延迟 2-3 秒再查一次进程列表，确认应用确实启动，提升反馈可靠性。
  4. 应用运行参数：shadowbot:Run? 协议是否支持传参（如指定运行模式）尚未探索。
